# Crown Games — Подписка — CLAUDE.md

Этот репозиторий содержит **только игру «Подписка»**. Один артефакт:

1. **`subscription`** — канистра ICP. Резолвер предоплаченных потоков формы `stream`: отдаёт дериватный ключ подписки, подписывает `release` созревшего по расписанию куска и `cancel` по Ed25519-подписи донора. **Состояния не имеет вовсе**: ни записей, ни таймеров, ни стабильной памяти. Денег не держит, транзакций не шлёт, внешние сети не читает, книгу не читает.

Контрактов здесь нет: игра работает на задеплоенной форме `stream` из `crown-factory` — по одному эскроу на подписку, резолвер дериватен от `subscription_id`. Ядро (`crown-core`) и фабрика (`crown-factory`) не трогаются ни одной строкой. Бот, мини-апп, привязки кошельков, политика каналов — `crown-bot-tg`, не здесь. См. `docs/post-game.md`.

Этот файл читается каждую сессию. Держи коротким.

## Незыблемые принципы

1. **Игра — это резолвер и только.** Дерево публичных ключей плюс два правила подписи: `release` — когда наступил срок куска, `cancel` — когда попросил донор. Write-канала «игра → ядро» не существует: единственное влияние на книгу — честный сеттлмент эскроу.
2. **Канистра без состояния.** Она не знает, какие подписки существуют: `subscription_id` — непрозрачные 32 байта derivation path, эскроу доказывается пересчётом адреса из предъявленных полей рождения в момент запроса. Нечего хранить — нечего потерять и нечего мигрировать.
3. **Порядок держит ончейн-форма, не канистра.** `index == released`, «не раньше срока», терминальность — инварианты формы `stream`; подпись канистры, не исполнимая ончейн, безвредна. Канистра проверяет только собственный закон (§4 спеки).
4. **Канистра не двигает деньги.** Ни транзакций наружу, ни ключей внешних сетей помимо threshold-подписи, ни нативного токена. Подпись исполняет кто угодно транзакцией `release`/`cancel`.
5. **Привязка эскроу к подписке — поле `RESOLVER`.** У каждого `subscription_id` свой дериватный Ed25519-ключ (`derivation_path = [subscription_id]`). `nonce` эскроу — произвольный u64 без смысловой нагрузки.
6. **Границы — компилятором, не договорённостью.** `logic` — крейт с нулём зависимостей; время — аргумент функции. Проверяется CI, не ревью.
7. **Авторизация — подпись кошелька, не вызывающий.** `get_resolver` и `request_release` бесправны: право даёт арифметика деривации и расписание. `request_cancel` требует Ed25519-подпись донора над сообщением, привязанным к `(canister_id, chain, escrow)`.
8. **Байты подписи — интерфейс, а не деталь.** message формы `stream` воспроизводится побайтово (game-spec §10); расхождение = деньги ждут `refund()`.
9. **Минимализм = безопасность.** У каждой строки есть текущий потребитель. Никакой обобщённости «на будущее». Сомневаешься между «добавить» и «удалить» — удаляй.

Расхождение кода с принципами или с `docs/` = баг. Не «решаем на месте»: правь код; если неверен принцип — сначала правь доку.

## Структура репозитория

```
subscription/
├─ CLAUDE.md
├─ docs/
├─ logic/                  # КРЕЙТ БЕЗ ЗАВИСИМОСТЕЙ. Расписание и закон release.
│  └─ src/{lib,schedule}.rs
├─ canister/               # КАНИСТРА. Ingress, деривация, подписи. Состояния нет.
│  ├─ subscription.did
│  └─ src/
├─ config/
│  ├─ testnet.toml
│  └─ mainnet.toml
├─ e2e/                    # Тестовая обвязка живых сетей (драйвер devnet-транзакций).
└─ scripts/
```

Зависимости строго в одну сторону: `canister → logic`, `canister → crown-derive` + `crown-salt` (внешние крейты из `crown-factory`, path-зависимости на working tree; ветки репозиториев согласуй). Обратных нет. `logic` не знает про ICP, про чейны, про криптографию, про Candid.

## Как работать

1. Прочитай `docs/build-plan.md`, определи текущий этап (первый не-`DONE`).
2. Работай строго по нему. Не забегай вперёд.
3. Этап не закрыт, пока Definition of Done не выполнен и тесты зелёные.
4. По завершении — обнови статус, удали устаревшую доку тем же коммитом, только потом следующий этап.

## Гардрейлы (NEVER)

- Никогда не добавляй в `logic` зависимость. Ни одну. Время и числа приходят аргументами; конверсия на границе в `canister`.
- Никогда не заводи в канистре состояние: ни стабильную память, ни heap-реестры, ни таймеры, ни кэши подписей. Канистра — чистая функция от аргументов и derivation path; апгрейд ничего не мигрирует, потому что мигрировать нечего.
- Никогда не давай канистре двигать деньги: ни `transfer`, ни `approve`, ни знания о сплиттере, казне, `feeBps`, флоре, ценах подписок.
- Никогда не читай внешние сети: ни RPC-канистры, ни `http_request`, ни `crown-index`. Единственный внешний вызов — management canister (threshold-ключи).
- Никогда не подписывай `release` раньше `t0 + index·period` по предъявленным полям рождения.
- Никогда не подписывай `cancel` без валидной Ed25519-подписи донора из предъявленных полей рождения, над сообщением game-spec §8.
- Никогда не подписывай вне точного байтового формата формы `stream` (game-spec §10) и вне derivation path `[subscription_id]`.
- Никогда не авторизуй действие по вызывающему principal.
- Никогда не добавляй update-метод сверх фиксированного списка в `.did`. Новый update = новая доверенная поверхность = сначала правка спеки.
- Никогда не хардкодь адреса, program id, principal-ы. Только `config/`.
- Никогда не трогай `crown-core` и `crown-factory`. Нужна другая форма или другие правила выпуска — это другой репозиторий.
- Никогда не добавляй код «на будущее» без текущего потребителя. Если хочется — это `docs/post-game.md`.

## Стек

- `logic`: Rust, zero deps, `#![forbid(unsafe_code)]`, только `checked_*` арифметика.
- `canister`: Rust, `ic-cdk`, `crown-derive`, `crown-salt`; криптография — только пиннутые библиотеки (`ed25519-dalek` или аналог), ничего самописного. `ic-stable-structures` не нужен — состояния нет.
- Тесты: property-тесты в `logic`, PocketIC для канистры, e2e-скрипт против devnet.
- Доки и проза — по-русски. Идентификаторы, комментарии, имена файлов кода — по-английски.
