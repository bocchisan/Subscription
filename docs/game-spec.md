# Подписка — Спецификация

**v1.0 · Канистра без состояния · Подпись по требованию**

**Класс B.** Холд предоплаты донора и начисление ему репутации по мере выпуска кусков через сплиттер. Один эскроу формы `stream` из `crown-factory` на одну подписку.

Контрактов в этой игре **ноль** (factory-spec §7): форма `stream` задеплоена, её фабрика пиннута в `factories[]` конфига ядра, игра лишь даёт клиентам дериватный резолвер и подписывает два вида сообщений формы. Ядро не изменяется. Фабрика не изменяется. Вся новизна — в канистре `crown-games/subscription`, и её меньше, чем в любой из двух предыдущих игр.

Канистра **без состояния**: она не хранит ни подписок, ни эскроу, ни вердиктов, не имеет таймеров и стабильной памяти. Оба её правила — чистые функции от предъявленных аргументов, времени и derivation path. Всё, что связывает эскроу с подпиской, доказывается деривацией: ключа — для членства, адреса — для полей рождения. Инварианты порядка (`index == released`, «не раньше срока», терминальность) держит сама ончейн-форма; канистра их не дублирует и дублировать не может — ей нечем помнить.

Читается вместе с `crown-core/docs/core-spec.md`, `crown-factory/docs/factory-spec.md` (форма `stream` — §2.4) и спеками двух предыдущих игр. Термины оттуда не переопределяются. «Подписка» — вырожденный резолвер: у «Заданий» и «Сбора» исход решает голосование, здесь — расписание и воля донора.

---

## 1. Что происходит

Владелец (получатель потока) получает у канистры резолвер для своего `subscription_id` и публикует его через своего клиента (бот, сайт — недоверенная зона, `crown-bot-tg`).

Донор рождает **один** эскроу формы `stream`: предоплата `chunk × n_chunks`, получатель — владелец, `RESOLVER` — ключ подписки. Одна транзакция, недофинансированного эскроу не бывает.

Дальше три пути, все — подпись по требованию:

- **Выпуск.** Кусок `index` созрел (`now ≥ t0 + index·period`) — кто угодно берёт у канистры подпись `release(index)` и исполняет её транзакцией. Владелец получает свою долю куска через сплиттер, донору капает репутация. Газ платит тот, кто жмёт, — обычно владелец, собирающий свои куски.
- **Отмена.** Донор подписывает кошельком сообщение отмены (§8), кто угодно относит его в `request_cancel` и исполняет `cancel` транзакцией: весь невыпущенный остаток немедленно возвращается донору. Терминально.
- **Страховка.** Если очередной кусок просрочен больше чем на `RELEASE_MARGIN` и его так и не выпустили — `refund()` формы возвращает донору весь остаток без всякой подписи. Против мёртвой канистры и ленивого сборщика.

Заработать больше внесённого нельзя никому. Приза нет. Автосписания не существует: продление = новый эскроу той же подписки.

## 2. Привязка эскроу к подписке

**`subscription_id` — непрозрачные 32 байта.** Канистра не создаёт и не регистрирует подписки: любой `subscription_id` валиден, его резолвер — дериватный Ed25519-ключ threshold-дерева канистры по пути `[subscription_id]` (`schnorr_public_key` / `sign_with_schnorr` с `derivation_path`). Смысл идентификатора — целиком забота клиента; рекомендация клиентам — доменный префикс в хеше (например, `sha256("crown:bot-tg:v1" ‖ …)`), чтобы два продукта на одной канистре не столкнулись ключами. Канистре всё равно.

**Членство — поле `RESOLVER`.** Донор рождает эскроу с `RESOLVER = resolver(subscription_id)`. Эскроу с чужим резолвером к подписке не деривуется; один эскроу к двум подпискам — тоже (два разных ключа). Это те же 32 байта криптографической стойкости, что и в «Сборе», в поле, которое форма и так проверяет на каждом `release`/`cancel`.

**Поля рождения — предъявляются, не хранятся.** На каждый запрос подписи предъявитель даёт поля рождения эскроу; канистра пересчитывает `salt` крейтом `crown-salt` и адрес крейтом `crown-derive` — той же арифметикой, что индексатор ядра, — подставляя `resolver` из деривации, никогда из аргументов. Неверная декларация даёт адрес, по которому эскроу не существует, — подпись для него бесполезна и безвредна: деньги двигает только сам эскроу.

`nonce` эскроу — произвольный u64 донора: разводит PDA двух одинаковых подписок, смысловой нагрузки не несёт (game-spec «Сбора» §2 — почему в `nonce` нельзя класть усечённые хеши; здесь это тоже верно и тоже не нужно).

## 3. Участники

| | Что это | Держит деньги | Может врать |
|---|---|---|---|
| Донор | человек (подписчик) | нет | — |
| Владелец | человек (получатель потока) | нет | — |
| **Игровая канистра** | канистра ICP, threshold-ключ, blackholed в проде; без состояния, подписывает по требованию | нет | нет* |
| **Эскроу** | программа формы `stream`, immutable | да, до расчёта | нет |
| **Сплиттер** | ядро | нет | нет |
| **crown-index** | ядро, книга | нет | нет |
| Клиент | `crown-bot-tg` (бот, мини-апп) | нет | **да** |

\* Для фабрики резолвер — недоверенный (factory-spec §8): границы его вранья — выпустить созревший кусок или отменить поток в пользу донора; украсть, перенаправить или ускорить он не может ничего. Игра сужает и это: код зафиксирован блэкхолом, правила — две чистые функции, приватного ключа не существует — подпись собирает подсеть ICP.

**`crown-index` в игре не участвует.** Голосования нет, веса не нужны — канистра книгу не читает вовсе. Репутация возникает у донора сама, стороной ядра, из `Settled` каждого выпущенного куска. Единственный внешний вызов канистры — management canister.

Клиент — вне денежного пути и вне пути подписи: его записи (кто подписан на какой канал, цены, привязки) — дисплей и членство в его собственной зоне, ничего в этой игре они не гейтят.

## 4. Закон подписи

Машины состояний нет — нечему переходить. Весь закон — два правила:

```
release(index) подписывается  ⇔  now ≥ t0 + index·period      (по предъявленным полям)
cancel         подписывается  ⇔  Ed25519-подпись донора валидна (по предъявленным полям, §8)
```

- Время — IC-время в момент запроса. Ончейн-форма проверяет срок ещё раз своими часами (`release` ревертится до `due`), так что расхождение часов не исполняет ничего раньше времени; проверка канистры — её собственный закон, а не дубль ончейн-гейта: резолвер не подписывает несозревшее.
- Порядок кусков канистра не знает и не проверяет: `index == released` держит форма. Подпись куска, чей черёд не настал, исполнится только когда настанет; подпись куска за пределами `n_chunks` не исполнится никогда.
- Оба правила детерминированы и не пишут ничего. Повторный запрос — та же подпись тех же байтов (threshold-подпись Ed25519 детерминирована по message и path) либо эквивалентная; исполнимость от повтора не меняется.
- Терминальность (`cancel`, последний `release`, `refund`) живёт в поле `settled` эскроу — ончейн. После неё любые подписи неисполнимы.

```
   ДОНОР ── create_escrow(владелец, chunk, n_chunks, t0, period,
     │                    resolver(subscription_id), nonce) ──► ФАБРИКА
     │                                                            │ рождает
     └── USDC: chunk × n_chunks ────────────────────────────► ЭСКРОУ

   кто угодно ── request_release(поля, index) ──► КАНИСТРА ── подпись, если срок настал
   кто угодно ── release(index, sig) ──► ЭСКРОУ ── piece(chunk, share_j) ──► СПЛИТТЕР
                                              │        └► владелец 97%, казна 3%,
                                              │           Settled(payer = эскроу)
                                              └── остаток куска → донор

   донор ── подпись отмены ──► request_cancel ──► КАНИСТРА ── подпись cancel
   кто угодно ── cancel(sig) ──► ЭСКРОУ ── весь невыпущенный остаток → донор. Терминально.

   кто угодно ── refund() ──► ЭСКРОУ ── остаток → донор,
                 строго после t0 + released·period + RELEASE_MARGIN, без подписи
```

## 5. Кнопки: транзакция, подпись или вызов

| Кнопка | Тип | Кто платит | Куда идёт |
|---|---|---|---|
| Узнать резолвер | **update-вызов**, бесправный | никто | канистра: `get_resolver` (§7) |
| Подписаться | **транзакция** | донор | фабрика: `create_escrow` (`RESOLVER` привязывает к подписке) |
| Запросить подпись выпуска | **update-вызов**, бесправный | никто | канистра: `request_release` (§7) |
| Выпустить кусок (`release`) | **транзакция** | кто нажал | эскроу `release(index, sig)` |
| Отменить (подпись) | **подпись** донора | никто | канистра: `request_cancel` (§7) |
| Отменить (исполнение) | **транзакция** | кто нажал | эскроу `cancel(sig)` |
| Страховка | **транзакция** | кто нажал | эскроу `refund()`, без подписи |

Ончейн-транзакций на путь одной подписки: одна — `create_escrow`, плюс по одной на выпущенный кусок и одна терминальная. Донор подписывает канистре **только отмену**; участие доказывает сама транзакция подписки.

Канистра не отправляет транзакций и не держит SOL. `get_resolver` и `request_release` — update-вызовы, не query: threshold-деривация и threshold-подпись — асинхронные вызовы management canister и из query невозможны; хранимой копии резолвера, которую могла бы отдавать query, у канистры нет — состояния нет (§6). Вызовы бесправные: право даёт арифметика.

| Что | Где | Кто может подделать |
|---|---|---|
| поля рождения эскроу: `DONOR, RECIPIENTS, SHARES, CHUNK, N_CHUNKS, T0, PERIOD, RESOLVER, NONCE` | нотаризованы адресом эскроу через `salt`; читаются из аккаунта эскроу | никто |
| `released`, `settled` | аккаунт эскроу, ончейн | никто |
| резолвер подписки | не хранится — деривация по требованию | никто |
| подписи release/cancel | не хранятся — выдаются по требованию | никто |
| привязки, цены, периоды, членство каналов | клиент (`crown-bot-tg`) | оператор клиента |

Канистра не хранит **ничего**. Нет записей — нет создания, нет листинга, нет миграции при апгрейде, нет сертифицированного состояния (сертифицировать нечего: оба ответа канистры проверяемы сильнее — подпись верифицируется против дериватного ключа, ключ детерминирован путём).

## 7. Подпись по требованию

Регистрации нет. Три бесправных update-метода:

**`get_resolver(chain, subscription_id)`** → `schnorr_public_key(derivation_path = [subscription_id])`, 32 байта Ed25519. Детерминирован; клиент зовёт при настройке и кэширует у себя.

**`request_release(chain, subscription_id, donor, recipients, shares, chunk, n_chunks, t0, period, nonce, index)`.** Канистра:

1. выводит `resolver = ключ пути [subscription_id]` — из деривации, не из аргументов;
2. собирает `salt = sha256(donor ‖ K ‖ recipients ‖ shares ‖ chunk_le ‖ n_chunks_le ‖ t0_le ‖ period_le ‖ resolver ‖ nonce_le)` крейтом `crown-salt` — побайтово кодировка формы `stream` (factory-spec §2.4);
3. выводит `escrow = PDA([b"escrow", salt], config.factory)` крейтом `crown-derive`;
4. проверяет закон: `now ≥ t0 + index·period` (checked-арифметика из `logic`; переполнение — отказ);
5. подписывает `message = DOMAIN ‖ program_id ‖ escrow ‖ [RELEASE_TAG, index_le]` через `sign_with_schnorr(derivation_path = [subscription_id])`, верифицирует результат против дериватного ключа и отдаёт `(escrow, index, подпись)`.

**`request_cancel(chain, subscription_id, поля рождения…, signature)`.** Шаги 1–3 те же; вместо закона времени — авторизация донора: Ed25519-подпись над сообщением §8 проверяется против `donor` **из предъявленных полей**. Валидна — канистра подписывает `message = DOMAIN ‖ program_id ‖ escrow ‖ [CANCEL_TAG]` тем же путём и отдаёт `(escrow, подпись)`.

Чужой резолвер невыводим по построению: он не аргумент. Эскроу, рождённый с другим резолвером, другим получателем или другой суммой, даст другой адрес — и его `release`/`cancel` отвергнут подпись. Канистра слепа и не знает, существует ли предъявленный эскроу в цепи; подпись для несуществующего адреса бесполезна и безвредна — деньги двигает только сам эскроу. Сожжённые на такой подписи циклы — принятое ограничение (§13.1).

Подмена `donor` в полях отмены не крадёт ничего: чужой `donor` — чужой `salt` — чужой адрес, по которому эскроу жертвы не живёт. Получить исполнимый `cancel` для реального эскроу может только тот, чьей подписью проверяется его реальное поле `donor`.

## 8. Авторизация отмены

Единственное действие с правом — отмена, право у донора. Ingress шлёт кто угодно; право даёт подпись кошелька, привязанная к `(canister_id, chain, escrow)`:

```
message = "crown:subscription:v1" ‖ lp(chain) ‖ lp(canister_id) ‖ lp(escrow) ‖ ACTION_CANCEL
```

`lp(x)` — длина u32 LE, затем байты (кодировка инъективна, как в «Сборе»); `ACTION_CANCEL = 0` — единственный байт действия, значение заморожено. `escrow` — адрес, который клиент донора выводит той же арифметикой соли заранее. Подпись Ed25519 Solana-кошельком над сырыми байтами.

Подпись из другой канистры, другого кластера или для другого эскроу невалидна. Повтор подписи безвреден: отмена терминальна ончейн, второй `cancel` ревертится на `settled`. Отзыва отмены нет — кто передумал, просто не исполняет транзакцию; неисполненная подпись отмены не мешает выпускам, но остаётся исполнимой до терминальности — донор, подписавший отмену, считает её совершённой.

## 9. Параметры

**Ручки донора.** `chunk`, `n_chunks`, `t0`, `period`, `nonce` — поля рождения его эскроу; их валидирует фабрика (`period > 0`, `n_chunks ≥ 1`, флор куска, представимость расписания — factory-spec §2.4). Что из них считать «ценой» и «месяцем» — политика клиента; канистра цен не знает.

**Констант правил нет.** Закон §4 не содержит ни одного числа, принадлежащего игре: `RELEASE_MARGIN` — константа деплоя формы (фабрика), цены и периоды — политика клиента.

**Конфиг-поверхность канистры.**

```toml
threshold_key = "key_1"                # имя threshold-ключа среды

[[chain]]
id      = "solana-devnet"              # кластер
factory = "<program id>"               # форма stream, из crown-factory deploy/
domain  = "crown:stream:solana-devnet" # входит в message подписи
```

Чего в конфиге нет: `crown_index` (книга игре не нужна), сплиттер, `feeBps`, казна, RPC-URL, `release_margin` (ончейн-константа формы, у канистры нет потребителя), цены, периоды, кворумы. Значения `factory`, `domain` берутся из `crown-factory/deploy/`; этот репозиторий их не порождает — только копирует. Чейны в конфиге попарно различны по `(id, factory, domain)` — иначе одна пара `(поля, path)` дала бы один адрес под двумя доменами; проверяется при деплое, как в «Сборе».

Init-оверрайдов нет: канистра не зовёт никого, кроме management canister.

## 10. Контрактная часть: готовая форма

Программа живёт в `crown-factory` и задеплоена на devnet; её инварианты доказаны там же (factory-spec §6, §2.4). Игра передаёт дериватный ключ подписки как `resolver` в каждый эскроу — и всё.

**Что подписывает канистра.** Единственный интерфейс игры с контрактом, побайтово:

```
release: message = DOMAIN ‖ program_id ‖ escrow ‖ [0x00, index u16 LE]
cancel:  message = DOMAIN ‖ program_id ‖ escrow ‖ [0x01]
```

`DOMAIN = "crown:stream:<id кластера>"` — вшит в задеплоенную форму, копируется в конфиг игры. Подпись Ed25519 (`sign_with_schnorr`, derivation path `[subscription_id]`). В транзакции подпись едет отдельной инструкцией `ed25519_program` непосредственно перед `release`/`cancel`; транзакцию собирает отправитель.

`DOMAIN` обязателен: кластеры делят program id. Подпись не переносится ни между кластерами, ни между эскроу, ни между действиями (теги `0x00`/`0x01` в одном фрейме), ни — из-за дериватных ключей — между подписками.

**Что форма гарантирует без канистры** (двигает деньги только она):

- куски строго по порядку и не раньше срока; каждый ненулевой кусок — через сплиттер (`Settled`, payer = эскроу), остаток куска — донору;
- `cancel` возвращает весь невыпущенный остаток донору, терминально;
- `refund()` — без подписи, строго после `t0 + released·period + RELEASE_MARGIN`, весь остаток донору, терминально;
- последний `release` терминален и закрывает ATA эскроу (рента — донору).

**Шов с ядром и репутация — не забота игры.** Индексатор сам читает у эскроу `salt` и `donor`, пересчитывает адрес от пиннутой фабрики и начисляет репутацию донору — по `piece`-суммам каждого выпущенного куска, по мере выпуска (core-spec §4). Репутация и комиссия возникают **только на `release`**; `cancel` и `refund()` идут мимо сплиттера.

**Что тестирует игра.** Инварианты эскроу — порядок, сроки, терминальность, чужая подпись, ноль двойных выпусков — доказаны в `crown-factory` и здесь не повторяются. Игра тестирует своё:

- message, собранный канистрой, побайтово принимается реальным контрактом: e2e на devnet — `create_escrow → release созревших кусков по подписям канистры → cancel остатка по подписи донора`; книга показывает донору `+Σ piece` выпущенного, аномалий ноль.
- Закон времени: `request_release` до срока — отказ; после — подпись; фаззом по `(t0, period, index, now)`, включая переполнения.
- Авторизация отмены: чужой ключ, чужой эскроу, чужая канистра, чужой кластер — отвергнуты; валидная подпись донора даёт исполнимый `cancel` — негативные тесты против реального контракта.
- Деривация: резолверы двух `subscription_id` различны и стабильны; адрес, выведенный канистрой, равен адресу реально рождённого эскроу.
- Подпись невалидна для эскроу другой подписки — негативный тест против реального контракта.
- **`git diff` в `crown-core` и `crown-factory` пуст.** Это и есть настоящий тест шва.

## 11. Централизованный слой

Клиент (`crown-bot-tg`: бот, мини-апп) — недоверенная зона со своей спекой. Для игры существенно одно: он не стоит ни в одном пути игры. `get_resolver`, `request_release`, `request_cancel` бесправны и доступны любому напрямую; транзакции подписывают кошельки донора и владельца. Клиент может врать только о своём: членство в его каналах, показ цен. Донор, не доверяющий клиенту, отменяет подписку без него.

## 12. Обновление игры

Канистра blackholed. Обновление — новая канистра; дерево threshold-ключей дериватно от `canister_id`, у новой все ключи другие сами по себе.

- Клиенты переключаются на новую канистру для новых подписок.
- Эскроу старой версии ссылаются на ключи её путей и разрешаются ею же; мигрировать нечего — состояния нет.
- Заблэкхоленная канистра не удаляется. При исчерпании циклов она замерзает; пополнить может кто угодно.
- Эскроу, чей резолвер замолчал, выходит через `refund()` по `RELEASE_MARGIN`.

Блэкхол — прод-шаг, зеркало S4 ядра и F6 фабрики: до выхода в прод канистра передеплоивается свободно.

## 13. Принятые ограничения

Не латать, а принять.

1. **Подпись в пустоту жжёт циклы.** Канистра подпишет валидно деривуемые поля для эскроу, которого нет в цепи, и `release` для `index ≥ n_chunks`, чей срок формально настал. Обе подписи неисполнимы и безвредны; threshold-подпись стоит циклов. Вектор спама принят; митигация — rate limit на boundary nodes и бюджет циклов, не код.
2. **Ленивый сборщик кормит `refund()`.** Кусок, не выпущенный за `RELEASE_MARGIN` после срока, делает весь остаток возвращаемым донору — включая куски за уже прожитые периоды. Это свойство формы и честный исход «недоставленное-незабранное возвращается плательщику»; напоминания сборщику — забота клиента, не игры.
3. **Подпись бессрочна.** Выданная подпись `release`/`cancel` исполнима, пока эскроу не терминален. Кэшировать, отзывать или ограничивать её по времени канистра не может — ей нечем помнить. Исполнимость гейтит ончейн-форма.
4. **Досрочного выхода дешевле `cancel` нет.** Возврат — тоже транзакция; донор платит сетевую комиссию за собственную отмену.
5. **Канистра не знает цен и политик.** «Эскроу достаточен для доступа к каналу X» — суждение клиента по полям рождения; игра лишь гарантирует, что поля не подделываемы (адрес нотаризует их).
6. **Время каждого правила — своё.** Канистра меряет закон IC-временем, форма — часами кластера. Подпись, выданная на границе секунды, может подождать исполнимости; ничего не исполняется раньше ончейн-срока.
7. **Донор с несколькими эскроу отменяет каждый отдельно.** Подписка ≠ аккаунт; отмена — свойство эскроу.
